是的，**现在这套规则对“生成”来说偏重了**。它很适合做最终交付和审核，但如果每次从剧本直接生成都完整吃一遍，速度肯定会慢。

核心问题不是规则错，而是现在 Generator 同时承担了太多事：

```text
生成分镜
+ 自我审稿
+ 题材识别
+ 项目设定继承
+ 物理连续性校验
+ Seedance 执行补丁
+ reviewer 式防错
+ 回归案例防错
```

其中有一部分应该留给 **reviewer**，不一定都压在 **generator** 里。

---

## 我的判断

现在规则可以分成三类：

| 类别                                |    是否该留在 Generator | 说明                 |
| --------------------------------- | -----------------: | ------------------ |
| 输出格式、cut_id、字段、镜头号                |            **必须留** | 这是解析合同             |
| 忠实剧本、人物称谓、命名状态                    |            **必须留** | 生成阶段不管住，后面很难修      |
| 组间承接、横屏左右关系                       |            **必须留** | 横屏核心能力             |
| 物理连续性核心规则                         |       **必须留，但要压缩** | 保留容器、门窗、道具归属几个关键点  |
| 生活动作链密度                           |        **必须留，但压缩** | 防止一镜塞太多动作          |
| 题材包详细说明                           |         **不该默认读全** | 只在明确题材时启用          |
| 项目包详细说明                           |         **不该默认读全** | 只在明确项目时启用          |
| Reviewer taxonomy / hard issue 细则 |  **不该放 Generator** | 放 reviewer 即可      |
| 大量示例、推荐/避免写法                      | **可以移出 Generator** | 示例越多，生成越慢          |
| regression cases                  |  **只给 Reviewer 用** | 不要让 Generator 常规读取 |

你现在的横屏 Generator 已经有“可选参考层”，题材包和项目包只在明确启用时使用，这是对的。
但实际执行时，如果 agent 还是把所有文件都读了，或者 Generator 本体太长，速度还是会慢。

---

## 最推荐的优化：拆成三档模式

不要只有一个“全量生产模式”。建议加一个很短的规则：

```md
## 生成质量档位

默认使用 `standard`。

- `fast`：用于快速初稿。只执行核心格式、剧本忠实、横屏调度、组间承接、基础物理、台词口型和资产控制。不主动读取题材包、项目包、回归案例。每组 Seedance 执行提示和运镜强化词控制在 1 句。
- `standard`：用于常规交付。执行完整 Generator 核心规则；只有剧本或任务明确指向时，才读取对应题材包或项目包。
- `strict`：用于最终交付或问题剧本。执行完整 Generator + 对应题材包/项目包，并在生成后交给 Reviewer 做完整审核。
```

工作流可以变成：

```text
普通生成：fast / standard
最终交付：standard + reviewer
高风险项目：strict + reviewer
```

这样大多数情况下不用每次都开最高规格。

---

## Generator 应该瘦身，Reviewer 保持重

我建议这样分工：

### Generator 只保留“生成时必须知道”的规则

保留这些核心：

```text
1. 输入模式判断
2. 输出格式合同
3. 忠实剧本 / 人物称谓 / 命名状态
4. 资产 9 个以内
5. 组间承接 / 第1组处理 / 承接状态来源
6. 横屏构图与方向
7. 物理核心：容器、门窗、道具归属、光源昼夜
8. 道具任务状态与阶段切换
9. 关键线索道具生命周期
10. 生活动作链密度
11. 镜头时长和台词容量
12. 特殊元素只在剧本出现时启用
13. 基础负面词
```

这些是生成阶段不管就容易出错的。

### Reviewer 负责重规则

这些可以主要放 Reviewer：

```text
1. P0 hard issue 细则
2. taxonomy
3. 审核证据要求
4. regression cases
5. 复杂题材专业检查
6. 语义交叉检查
7. source_missing / technical_review_only
```

Reviewer 慢一点没关系，因为它不是每次都要重新创作全文。

---

## 具体可以怎么瘦

### 1. 把 Generator 里的长示例删掉或移到 Reviewer

Generator 里现在有很多“推荐 / 避免”的写法。示例对质量有帮助，但会拖慢生成。

可以保留每类最多一个短例子，其余移到 reviewer 或 regression cases。

比如“承接状态来源”不用写很多例子，Generator 里可以压成：

```md
组间承接只能引用上一组镜头或组尾已建立的状态。未建立的危险、道具或线索，必须写成本组首次揭示。
```

详细反例放 reviewer。

---

### 2. 题材包不要默认读

Topic Packs 现在覆盖很多题材，确实有价值。
但如果每次都读全，肯定慢。

建议在 workflow 里写死：

```text
只有当用户明确指定题材，或剧本强烈指向某题材时，才读取对应题材包。
一次最多启用 1 个主题材包 + 1 个辅助题材包。
不得默认读取全部 TOPIC_PACKS.md。
```

例如：

```text
豪门短剧 -> 只启用都市情感 / 豪门短剧
古装宫斗 -> 只启用古装权谋
雨龙吟端午 -> 启用乡村奇幻 + yulongyin 项目包
```

不要所有题材都扫。

---

### 3. 项目包只在项目明确时读

你现在已经有项目包机制。
建议再加一句：

```md
如果任务未明确指定项目包，不读取 project_packs/ 下任何文件。
如果任务明确指定项目，只读取对应项目包，不读取其他项目包。
```

这可以避免后续项目多了以后越来越慢。

---

### 4. `运镜强化词` 和 `Seedance执行提示补充` 限长

现在这两个字段很有用，但也容易让模型写很长，拖慢输出。

可以加：

```md
`运镜强化词` 控制在 1-2 句，只概括本组镜头策略，不重复镜头描述。
`Seedance执行提示补充` 控制在 1-2 句，只写最容易生成错的站位、道具、物理或特殊元素，不复述剧情。
```

现在很多组已经写得不错，但规则里限制一下会更稳。

---

### 5. 负面词可以考虑“继承式写法”

如果你们下游允许，不要每组完整重复一大串 `--neg`。
可以改成：

```text
**全片基础--neg**：
模糊，低分辨率，扭曲……

每组：
**--neg**：继承全片基础负面词，补充：夸张魔法特效，科幻激光感……
```

这样输出会短很多，生成也会快。

但如果 Seedance / 解析器要求每组独立可投喂，那就继续保留每组完整 `--neg`。这个取决于你们的自动化链路。

---

### 6. 输出格式可以保留，但字段内容更短

现在每组字段很多，格式合同不建议删。
但可以限制内容密度：

```md
**组间承接**：1-2 句。
**横屏构图/调度**：1-2 句。
**光影设计**：每镜 1 句。
**运镜强化词**：1-2 句。
**Seedance执行提示补充**：1-2 句。
```

这不会牺牲解析稳定性，但能明显减少输出长度。

---

## 我建议的轻量版 Generator 结构

可以把 `storyboard-horizontal-generator/SKILL.md` 改成更短的核心版：

```text
1. 使用范围和可选参考层
2. 输入模式判断
3. 最高优先级
4. 输出格式合同
5. 角色与设定一致性
6. 资产规则
7. 组间承接：第1组、同空间、换空间、承接状态来源
8. 横屏构图与方向
9. 物理核心：容器、门窗、道具归属、光源
10. 道具任务状态与阶段切换
11. 关键线索道具生命周期
12. 生活动作链密度
13. 镜头时长与台词容量
14. 特殊元素通用登记
15. 台词、声音与表演
16. 负面词
17. 禁止项
18. 生成步骤
```

但是每节都压短。
把详细案例、反例、taxonomy、回归点都留给 reviewer。Reviewer 本来就很重，它适合做门禁。

## 推荐的实际工作流

### 快速生成

```text
generator fast
不启用 topic pack
不启用 project pack
不跑 reviewer
```

用于看整体剧情和分组。

### 常规生产

```text
generator standard
只启用明确相关 topic/project pack
生成后跑 reviewer
只修 reviewer 报错组
```

### 最终交付

```text
generator strict
对应项目包 + 对应题材包
reviewer full audit
必要时局部重生 failed groups
```

## 最小改动方案

如果你不想大改，我建议只加这几条：

```md
## 性能模式

默认 `standard`。用户说“快速生成 / 先出一版 / 粗稿”时使用 `fast`。

fast 模式：
- 不主动读取 TOPIC_PACKS.md；
- 不主动读取 project_packs/；
- 每组镜头数优先 2-4 个；
- 组间承接、横屏构图、运镜强化词、Seedance执行提示各控制在 1-2 句；
- 只执行核心规则：格式、剧本忠实、横屏方向、组间承接、资产、物理、台词口型、特殊元素克制；
- 复杂审稿交给 reviewer，不在生成阶段展开长自检。

standard 模式：
- 执行完整 Generator 核心；
- 只读取明确相关的题材包或项目包；
- 生成后建议跑 reviewer。

strict 模式：
- 用于最终交付；
- 执行完整 Generator + 相关题材包/项目包；
- 必须跑 reviewer。
```

这条非常实用，可以先加。

## 结论

**是的，现在规则偏重。**
但不要直接删掉物理、承接、道具生命周期这些核心规则，因为它们确实修掉了 EP02 / EP03 的关键问题。

最优方案是：

```text
Generator 变轻：只负责正确生成。
Reviewer 保持重：负责完整审稿。
Topic / Project packs 按需读取。
增加 fast / standard / strict 三档。
限制每个字段句数。
```

这样速度会明显改善，而且不会把好不容易建立起来的质量门槛打掉。
